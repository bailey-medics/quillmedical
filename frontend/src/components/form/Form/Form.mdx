# Form

The `<Form>` component is the **standard wrapper for all single-page forms** in the application. It owns the submission lifecycle — loading state, success/error feedback, timeout handling — so individual pages don't need to reimplement this boilerplate.

## When to use

Use `<Form>` by default for any page or modal that collects data and submits it to an API.

This includes:

- Admin settings pages
- User profile / account forms
- Login, registration, forgot password
- Patient data entry
- Messaging compose flows
- Any single-page save/send workflow

## When NOT to use

- **Multi-step forms** — use `MultiStepForm` instead. It has its own step navigation, per-step validation, and confirmation step pattern that don't fit the `<Form>` model.
- **Teaching assessments** — these have timer-driven flows, multi-question navigation, and scoring logic that require custom state management.
- **Inline toggles** — single-field saves (e.g. a live status switch) where a full FormStatus card would be overkill.

## Four submission patterns

Form supports four submission patterns — confirm is orthogonal to stay/navigate:

| #   | Pattern                    | Components                                   | Feedback location               |
| --- | -------------------------- | -------------------------------------------- | ------------------------------- |
| 1   | Submit, stay on page       | `<Form>` + `<FormStatus>`                    | Status card on same page        |
| 2   | Submit + confirm, stay     | `<Form>` + `<ConfirmModal>` + `<FormStatus>` | Status card on same page        |
| 3   | Submit, navigate away      | `<Form>` + `PageMessage` (centralised)       | Status card on destination page |
| 4   | Submit + confirm, navigate | `<Form>` + `<ConfirmModal>` + `PageMessage`  | Status card on destination page |

## Architecture

Form manages a state machine with seven states:

```
idle → submitting → success
                  → partial_success
                  → error
                  → timeout
     → validation_error (client-side validation failed)
```

- **idle** — no submission attempted
- **submitting** — request in flight (button shows spinner + submitting label)
- **success** — full success
- **partial_success** — composite write where some layers succeeded (e.g. letter saved but GP delivery pending)
- **error** — server rejected or handler threw
- **validation_error** — client-side validation failed; uses amber/warning severity to avoid alert fatigue. Shows "Please check the highlighted fields" with a count of fields needing attention
- **timeout** — request exceeded the configured timeout (default 30 seconds)

The state is exposed to child components via React context. `FormStatus` and `SubmitButton` subscribe to this context automatically — no prop threading needed.

## Basic usage (§2 — stay on form)

```tsx
import { Form, FormStatus, SubmitButton } from "@components/form/Form";
import { TextField, SelectField } from "@components/form";

function SettingsPage() {
  const handleSubmit = async (data) => {
    await api.put("/api/settings", data);
    return {
      state: "success",
      message: { title: "Settings saved" },
    };
  };

  return (
    <Form
      defaultValues={{ name: "", role: "staff" }}
      onSubmit={handleSubmit}
      submitLabel="Save"
      submittingLabel="Saving…"
    >
      <Stack gap="md">
        <FormStatus />
        <TextField label="Name" />
        <SelectField label="Role" data={["staff", "admin"]} />
        <SubmitButton />
      </Stack>
    </Form>
  );
}
```

## Destructive actions (§1 — confirm modal)

For destructive or irreversible actions, use the `confirm` prop on `<Form>` to gate submission behind a confirmation modal. The modal manages its own async loading state — the form stays idle until the user confirms.

```tsx
import { IconAlertTriangle } from "@tabler/icons-react";

function PermissionsPage() {
  const handleSubmit = async () => {
    await api.delete("/api/permissions/123");
    return {
      state: "success",
      message: { title: "Permissions deleted" },
    };
  };

  return (
    <Form
      onSubmit={handleSubmit}
      submitLabel="Delete permissions"
      submittingLabel="Deleting…"
      confirm={{
        title: "Delete system permissions",
        acceptLabel: "Delete",
        submittingLabel: "Deleting…",
        icon: <IconAlertTriangle />,
        children:
          "This will permanently delete all permissions. This action cannot be undone.",
      }}
    >
      <Stack gap="md">
        <FormStatus />
        {/* ... fields ... */}
        <SubmitButton />
      </Stack>
    </Form>
  );
}
```

### ConfirmModal features

- `onAccept` can be sync or async — spinner shows on the accept button while the promise is pending
- Both buttons disabled during loading (prevents double-submit)
- Button width is locked during loading to prevent layout shift
- On success: calls `onClose` automatically, loading state persists through the close animation (no flash)
- On error: stays open, caller handles error display
- `closeOnClickOutside={false}` — prevents accidental dismissal
- No close button — forces conscious dismissal via Cancel or confirm

## Navigate away on success (§3)

For forms that navigate to a different page on success, pass a flash payload via React Router state. The centralised `PageMessageProvider` (in `MainLayout`) picks it up automatically and displays the message on the destination page — no per-page component needed.

```tsx
import { useNavigate } from "react-router";

function SendLetterPage() {
  const navigate = useNavigate();

  const handleSubmit = async (data) => {
    const result = await api.post("/api/letters", data);
    navigate("/patients/123/letters", {
      state: {
        flash: {
          title: "Letter sent",
          description: `ID #${result.id}`,
        },
      },
    });
    return { state: "success", message: { title: "" } };
  };

  return (
    <Form
      onSubmit={handleSubmit}
      submitLabel="Send letter"
      submittingLabel="Sending…"
    >
      <Stack gap="md">
        <FormStatus />
        {/* ... fields ... */}
        <SubmitButton />
      </Stack>
    </Form>
  );
}
```

On the destination page, the message appears automatically — no extra component needed.
The `PageMessageProvider` reads `location.state.flash` and renders it above the page content.

For imperative page-level messages (e.g. after a delete action outside a `<Form>`), use
the `usePageMessage` hook:

```tsx
import { usePageMessage } from "@components/page-message";

function OrganisationsPage() {
  const { showMessage } = usePageMessage();

  async function handleDelete(org) {
    await api.del(`/organizations/${org.id}`);
    showMessage({
      variant: "success",
      title: "Organisation removed",
      description: `${org.name} has been removed`,
    });
  }

  return (
    <Container size="lg" py="xl">
      {/* PageMessageDisplay is rendered by MainLayout — no import needed */}
      <OrgList onDelete={handleDelete} />
    </Container>
  );
}
```

## Components

### `<Form>`

The outer wrapper. Accepts `defaultValues`, an async `onSubmit` handler that returns a
`FormSubmitResult`, and optional `submitLabel`, `submittingLabel`, `timeoutMs`,
`disableWhenClean`, and `confirm` props.

When `disableWhenClean` is true, the submit button is disabled until the user changes a field value.

When `confirm` is provided, a `<ConfirmModal>` gates submission — the form validates first, then shows the modal. On confirm, the `onSubmit` handler runs.

### `<FormStatus />`

Displays a status card when the form is in a result state (success, error, partial_success, timeout, validation_error).
Renders nothing during idle/submitting. Place it at the top of your form layout.

- Uses `role="status"` for success/partial success, `role="alert"` for errors/timeout/validation
- Scrolls the page to the top when it appears
- Dismissible by the user, never auto-dismissed
- Can also be used standalone with manual props: `<FormStatus variant="success" title="Done" />`

### `<SubmitButton />`

Subscribes to form context for label and loading state. Uses `aria-disabled` (not `disabled`)
so it remains focusable for screen readers during submission. Shows a spinner alongside the
submitting label during submission, with the button width locked to prevent layout shift.

Optional `onCancel` prop adds a cancel button in a ButtonPair-style layout.

### `<ConfirmModal />`

A standalone confirmation modal for destructive or irreversible actions. Lives in
`components/confirm-modal/` and can be used both inside and outside `<Form>`.

Props: `opened`, `onClose`, `onAccept` (async-aware), `children`, `title?`, `acceptLabel?`,
`submittingLabel?`, `cancelLabel?`, `icon?`.

## Handling errors

Your `onSubmit` handler should return a `FormSubmitResult` for both success and expected errors:

```tsx
const handleSubmit = async (data) => {
  const res = await api.put("/api/settings", data);
  if (!res.ok) {
    return {
      state: "error",
      message: {
        title: "Failed to save",
        description: "The server returned an error. Please try again.",
      },
    };
  }
  return {
    state: "success",
    message: { title: "Settings saved" },
  };
};
```

If your handler throws an unexpected error, Form catches it and shows a generic error message.
If the request exceeds `timeoutMs`, Form aborts and shows a timeout message automatically.

## Partial success

For composite writes (e.g. saving a letter and sending to a GP), use `partial_success`
when some operations succeeded but others failed:

```tsx
return {
  state: "partial_success",
  message: {
    title: "Partially saved",
    description:
      "Letter saved to patient record. Delivery to GP is still pending.",
  },
};
```

## Field components

All existing form field components work inside `<Form>` as children:
`TextField`, `SelectField`, `TextAreaField`, `MultiSelectField`, `PasswordField`,
`SolidSwitch`, `RadioField`.
